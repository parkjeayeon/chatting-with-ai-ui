## 0. 한눈에 보는 최종 결론

| 레이어 | 최종 선택 | 한 줄 이유 |
| --- | --- | --- |

## 1. 배경 — 왜 "다 비슷하게 생겼나"

레퍼런스 사이트들이 닮은 이유는 공통 보일러플레이트가 있어서가 아니라, **같은 재료**를 쓰기 때문이다.

- Next.js (React) + Tailwind CSS — 사실상 표준
- shadcn/ui 또는 Radix UI — "AI 사이트 룩"의 대부분
- Vercel AI SDK — 스트리밍 채팅의 사실상 표준
- Framer Motion, Inter/Geist 폰트

또한 이 사이트들은 사실 **두 개의 다른 화면**을 합쳐 놓은 구조다.

| 구분 | 내용 | 제작 방식 |
| --- | --- | --- |
| (A) 마케팅/랜딩 | "Request a demo", 로고, 제품 소개 | Webflow 또는 Next.js 템플릿 |
| (B) 제품 앱 | 로그인 후 채팅/리서치 화면 | 에이전틱 리서치 UI (본체) |

> Causaly 홈은 실제로 `website-files.com` 도메인 = **Webflow**로 제작됨.
> 우리가 만들 핵심은 **(B)** 이며, (B)들이 닮은 건 동일한 "에이전틱 리서치 UI 패턴"을 따르기 때문이다.

### 제품 앱 (B)의 공통 4구역

1. 중앙 채팅 입력 — "질문 = 리서치 시작"
2. 에이전트 작업 과정 패널 — Plan → Search → Read → Reason → Conclude (OpenManus가 채움)
3. 근거/인용 패널 — 출처 논문/문서 카드 (일반 ChatGPT와 결정적 차이)
4. 세션/히스토리 사이드바

## 2. 결정 ① — UI 라이브러리

### 선택지

| 후보 | 설명 | 장점 | 단점 |
| --- | --- | --- | --- |
| Vercel AI Chatbot 템플릿 | Next.js + AI SDK + shadcn/ui | ChatGPT 스타일 즉시 제공, 표준에 가까움 | 단순 말풍선 위주, 에이전트 과정 표현 약함 |
| **assistant-ui** | 채팅 인터페이스 전용 React 라이브러리 | 스트리밍·툴콜·인용·스텝 렌더링 내장, 조합형 primitive | 빠르게 바뀌는 라이브러리 |
| chatbot-ui / LibreChat | 오픈소스 ChatGPT 클론 / 멀티모델 풀스택 | 빠른 시작 | 에이전트 과정 시각화엔 부적합 |

### 결정: **assistant-ui**

이유: 사이언스 플랫폼의 핵심 차별점은 "AI가 무엇을 **하고 있나**"를 보여주는 것. 단순 말풍선 UI로는 부족하고, assistant-ui는 Thread·Message·Composer·ThreadList·ActionBar 같은 조합형 primitive로 어떤 UX든 구성 가능하며 에이전트 스텝/툴콜/인용 렌더링이 내장돼 있다.

## 3. 결정 ② — 프론트–백엔드 연결 방식 (Runtime)

assistant-ui는 백엔드 연결 방식(runtime)을 골라야 한다. OpenManus는 **기성 프레임워크 어댑터가 없으므로** 커스텀 경로 중 하나를 택해야 한다.

### 선택지 (assistant-ui 공식 4경로)

| 경로 | 기반 | 와이어 형태 | 선택 기준 |
| --- | --- | --- | --- |
| LocalRuntime | Core | `ChatModelAdapter.run` 작성 | 단순 fetch 한 번, 상태 관리는 라이브러리에 위임 |
| ExternalStoreRuntime | Core | messages + 콜백 제공 | 이미 redux/zustand 등에 상태 보유 |
| DataStream | LocalRuntime + 프로토콜 | 백엔드가 data stream 프로토콜 방출 | 얇은 메시지 스트림 계약만 필요 |
| **AssistantTransport** | ExternalStoreRuntime + 프로토콜 | 백엔드가 **에이전트 상태 스냅샷** 스트리밍 | 에이전트 내부 상태가 풍부하고 UI에 드러내고 싶을 때 |

### 대안 고려: AG-UI 프로토콜

OpenManus를 AG-UI 표준 이벤트(thinking/reasoning/tool call/state snapshot)로 래핑하는 방법도 있다. 표준이라 깔끔하지만 OpenManus를 AG-UI 형식으로 내보내도록 감싸는 추가 작업이 필요하다.

### 결정: **AssistantTransport**

이유:
- 공식 문서가 "커스텀 에이전트 프레임워크 또는 스트리밍 프로토콜이 없는 프레임워크(예: 오픈소스 LangGraph)"에 이 경로를 권장 — OpenManus가 정확히 이 경우.
- 메시지 조각이 아니라 **전체 에이전트 상태의 스냅샷**을 스트리밍 → Plan/Search/Read/Reason 같은 과정을 그대로 화면에 미러링 가능.
- `controller.state` 딕셔너리를 수정하면 diff가 자동 전송됨 (`set` / `append-text` 두 연산).
- 사이언스 플랫폼에 필요한 steps·citations를 state에 같이 담아 사이드 패널과 채팅 본문을 한 상태로 동기화.

## 4. 확정 아키텍처

```
[랜딩]  Next.js + Tailwind + shadcn/ui  (또는 Webflow)

[제품 앱]  Next.js + assistant-ui + shadcn/ui
            │  useAssistantTransportRuntime + converter
            ↕  POST(commands) / stream(aui-state:...)
[백엔드]  FastAPI + assistant-stream (create_run / RunController)
            ↕
        OpenManus 에이전트 루프
            └ plan / tool_call / tool_result / token 이벤트를
              controller.state 의 messages·steps·citations 로 미러링
```

### 만들어야 하는 것은 딱 2개

1. **백엔드 엔드포인트** — 커맨드를 받아 상태 스냅샷 스트림 반환
2. **프론트 상태 컨버터** — 상태 스냅샷을 assistant-ui 메시지 포맷으로 매핑

추가로 사이언스 플랫폼 고유의 **스텝 패널 / 근거 패널**은 `useAssistantTransportState`로 state를 직접 읽어 렌더링한다.

## 5. 핵심 엔지니어링 포인트

> UI 디자인보다 **OpenManus의 내부 이벤트를 어떤 state 스키마(steps/citations 모양)로 미러링할지** 설계하는 것이 품질을 좌우한다.

- state 스키마가 정해지면 백엔드 어댑터와 프론트 패널이 동시에 확정된다.
- 취소 처리: OpenManus 루프 조건에 `and not controller.is_cancelled` 포함.
- 토큰 스트리밍: 최종 답변은 `state["messages"][-1]["content"] += event.text` → 자동 `append-text`.
