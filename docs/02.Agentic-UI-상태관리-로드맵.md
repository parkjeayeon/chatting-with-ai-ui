# Part 1. 설계 제1원칙

## "무엇을 답했나"가 아니라 "무엇을 하고 있나"

노트의 핵심 명제를 그대로 설계 원칙으로 승격한다.

> 기존 UI는 **AI의 답변(결과)만** 보여주면 됐다.
> Agentic UI는 **AI가 실제 작업을 수행하는 "과정" 전체**를 보여줘야 한다.
> → "무엇을 답했나"가 아니라 **"무엇을 하고 있나"**를 화면에 드러내야 한다.

이 원칙이 아키텍처에 주는 직접적 함의는 다음과 같다.

1. **UI는 무상태(stateless) 뷰여야 한다.** 화면이 자체적으로 상태를 만들지 않고, 백엔드(OpenManus)가 흘려보내는 "에이전트 상태"를 그대로 비춘다. → 이것이 AssistantTransport를 택한 근본 이유.
2. **상태는 "결과"가 아니라 "과정"을 담아야 한다.** 따라서 상태 트리에 `messages`만 두면 안 되고, `plan`, `steps`, `toolCalls`, `citations`, `approvals`, `actionLog`가 1급 필드로 들어가야 한다.
3. **모든 진행은 추적 가능(traceable)해야 한다.** 사용자가 "멈춘 건지/진행 중인지", "어디서 실패했는지", "무엇을 승인 대기 중인지"를 항상 알 수 있어야 한다.

## 기존 UI vs Agentic UI — 상태 복잡도의 근본 차이

노트의 비교를 상태관리 언어로 옮기면:

| 구분 | 기존 UI (예: 매출 대시보드) | Agentic UI |
| --- | --- | --- |
| 상태의 출발점 | **사용자의 클릭·입력** | **사용자의 목표 + AI의 자율 실행** |
| 상태 전이 | 사용자 조작에 1:1 반응 | AI가 단계를 스스로 구성 → 전이가 비결정적 |
| 상태 종류 | `idle → loading → success/error/empty` | 수십 개 (planning, plan_ready, waiting_for_approval, tool_calling, streaming …) |
| 화면 가정 | 고정된 순서 | **고정된 순서를 가정할 수 없음** |
| 신뢰의 근거 | 결과값 | 결과 + 과정(계획·도구·근거·실패) |

> 일반 자동화 UI와의 결정적 차이: 일반 자동화는 "버튼 → 미리 정해진 순서". Agentic UI는 "목표 → AI가 순서를 그때그때 구성". 그래서 화면도 **단계 종류별 재사용 가능한 공통 구조**로 만들어야 한다.

# Part 2. 핵심 결정

## 왜 상태가 폭발하는가

노트가 지적한 대로, `idle · loading · success · error` 4개로는 표현이 불가능하다. 최소한 다음이 필요하다.

```text
idle
→ prompt_submitted      (목표 제출됨)
→ planning              (AI가 계획 수립 중 — 외부 호출 없음)
→ plan_ready            (계획 완성, 미리보기)
→ waiting_for_approval  (위험 작업, 승인 대기)
→ tool_calling          (도구 실제 호출 중 — 외부 I/O)
→ streaming             (결과 생성 중)
→ partial_failure       (일부 단계 실패, 부분 결과 가능)
→ completed / cancelled / error
```

## 왜 AssistantTransport인가 (이 맥락에서 재확인)

상태가 이렇게 풍부해지면, "메시지 조각만 스트리밍"하는 방식(DataStream)으로는 부족하다. 우리에게 필요한 것은 **에이전트의 내부 상태 전체를 스냅샷으로 흘려보내고, UI가 그 위에 얹히는** 구조다. AssistantTransport가 정확히 이것을 제공한다.

- 백엔드는 `controller.state`(임의의 JSON)를 수정한다 → `set` / `append-text` 연산으로 **자동 diff 전송**.
- 프론트는 `converter`로 그 상태를 assistant-ui 메시지로 변환하고, 동시에 `useAssistantTransportState`로 **스텝·근거·승인 패널이 같은 상태를 직접 구독**한다.
- 승인 같은 사용자 액션은 **custom command**로 백엔드에 되돌려, OpenManus 루프를 재개시킨다.

> 한 줄 요약: **"AI의 자율 실행을 사람이 이해·통제할 수 있게" 만드는 9개 장치 = 9개 상태 슬라이스**. AssistantTransport는 이 9개 슬라이스를 한 상태 트리로 묶어 양방향으로 동기화하는 수단이다.

# Part 3. 단일 상태 트리(State Tree) 전체 설계

모든 패널은 **하나의 상태 트리**에서 파생된다. 아래가 `controller.state`(백엔드) ≡ `ExternalState`(프론트)의 전체 형상이다.

```jsonc

  "threadId": "thr_abc123",
  "status": "tool_calling",          // Part 5의 상태 머신 값
  "messages": [
    {
      "id": "msg_1",
      "role": "user",
      "content": "지난달 매출 하락 원인 분석해서 리포트로 정리해줘",
      // goal 필드는 최상위 스냅샷에만 보관하여 이중화 방지
    },
    {
      "id": "msg_2",
      "role": "assistant",
      "content": "초안 분석 결과입니다 …",
      "resultStatus": "partial"      // 4.6 draft|partial|verified|final
    }
  ],

  // ── 1. Goal Input ──────────────────────────
  "goal": {

    "planId": "plan_v1",
    "toolCallIds": ["tc_1", "tc_2", "tc_3"],
```

> 핵심 설계 결정: **9개 단계 + 근거(Citations) = 10개 최상위 슬라이스**. 각 슬라이스는 서로 ID로 참조(`planRef`, `toolCallId`, `stepRef`, `approvalId`)하여 느슨하게 결합한다. 이렇게 하면 한 패널의 변경이 다른 패널을 깨지 않으면서도, 타임라인↔툴콜↔근거↔승인이 한 상태로 일관되게 묶인다.

# Part 4. 9단계 × 상태 매핑

각 단계를 (노트 내용 보존) → (상태 슬라이스) → (백엔드 미러링) → (프론트 렌더링) 순으로 상세화한다.

## 4.1 Goal Input

### 노트 내용
사용자의 목표를 입력받는 첫 단계. 자연어 한 줄도 가능하지만 **자연어 + 구조화된 정보(범위·제약)**를 함께 받으면 훨씬 효과적. 수집 대상: 목표 · 작업 범위 · 사용할 데이터 · **금지할 행동** · 출력 형식 · 승인이 필요한 작업.

| 항목 | 예시 |
| --- | --- |
| 목표 | 지난달 매출 하락 원인 분석 |
| 범위 | US West 지역 |
| 데이터 소스 | sales DB, marketing DB |
| 출력 형식 | 표, 요약 |
| 실행 권한 | 조회는 자동 / 외부 발송은 승인 필요 |

> 프론트엔드 관점 전환: 단순 "채팅창"이 아니라 **목표와 작업 조건을 수집하는 구조**로 설계.

### 상태 매핑
`messages[].goal` 또는 최상위 `goal` 슬라이스. 위 표의 6개 항목이 `objective / scope / dataSources / outputFormat / forbiddenActions / permissions`에 대응.

### 백엔드 미러링 (OpenManus)
사용자 메시지가 `add-message` 커맨드로 들어오면, 백엔드는 목표를 파싱하여 `controller.state["goal"]`을 `set` 한다. OpenManus의 시스템 프롬프트/도구 권한도 이 `permissions`에서 파생.

### 프론트 렌더링
첫 화면을 "Goal Composer"로 구성: 자연어 입력 + (선택) 범위/데이터소스/출력형식 칩 + 권한 토글. **금지 행동(forbiddenActions)**은 빨간 영역으로 명시.

## 4.2 Plan Preview

### 노트 내용
실행 **전에** AI의 계획을 보여주는 단계. AI가 곧바로 실행하면 사용자는 불안을 느낀다. 특히 여러 데이터 조회·외부 시스템 변경 시 실행 전 계획 제시 필수. 사용자는 계획을 보고 **수정** 가능.

계획이 안전장치인 이유:

| 시점 | 계획이 해주는 일 |
| --- | --- |
| 실행 전 | AI가 무엇을 하려는지 확인 |
| 실행 중 | 지금 어느 단계까지 왔는지 추적 |
| 실행 후 | 어떤 계획에 따라 결과가 나왔는지 역추적 |

> 계획이 없으면: "이 결론은 어디 데이터에서?", "왜 이 API를?", "왜 이 범위까지?" → **결과뿐 아니라 계획도 UI/UX의 일부**.

### 상태 매핑
`plan` 슬라이스. `plan.status`(drafting→ready→approved→revised), `plan.editable`, `plan.steps[]`. 실행 중에는 각 `steps[].planRef`로 타임라인과 연결되어 "실행 중 추적"을, 실행 후에는 `audit.planId`로 "역추적"을 제공한다 → 노트의 3시점(전/중/후)을 모두 한 슬라이스로 커버.

### 백엔드 미러링
OpenManus가 plan 단계를 생성하면 `state["plan"]["steps"]`를 `set`. 계획 수립 동안 `status="planning"`(외부 호출 없음 → Part 5에서 `tool_calling`과 구분). 계획 완성 시 `plan.status="ready"`.

### 프론트 렌더링
"Plan Preview" 카드: 단계 리스트 + (editable이면) 각 단계 편집/삭제/추가 + [실행] 버튼. 위험 작업이 포함되면 실행 버튼 옆에 승인 필요 배지.

> 구현 노트: Plan 수정은 AssistantTransport의 **custom command**(`revise-plan`)로 보낸다. 백엔드가 plan을 갱신하고 `revised`로 표시.

## 4.3 Agent Timeline

### 노트 내용
AI가 **지금 무엇을 하는지** 단계별로 보여주는 영역. "로딩 중", "생각 중"만 보여주면 사용자는 아무것도 알 수 없다.

```text
✅ 요청 분석 완료
✅ sales 데이터 조회 완료
⏳ 지역별 매출 변화 계산 중
⬜ 원인 후보 정리 (대기)
⬜ 리포트 초안 작성 (대기)
```

이점: 멈춘 건지/진행 중인지 구분, 실패 위치 파악, **재시도 시 전체가 아니라 실패 지점부터**.

### 상태 매핑
`steps[]` 슬라이스. `status`(pending|running|done|failed|skipped) + `startedAt/endedAt` + `planRef`(계획 연결) + `toolCallId`(도구 연결).

### 백엔드 미러링
단계 진입 시 `steps`에 항목 append + `status="running"`. 도구 호출이 끝나면 `status="done"`, `endedAt` 기록. 실패 시 `status="failed"` + `failures[]`에 항목 추가.

### 프론트 렌더링
좌측 또는 중앙 상단 "Timeline": 아이콘(✅/⏳/⬜/❌) + 라벨 + 소요시간. `running` 단계는 `tw-shimmer` 등으로 진행감 표시. 실패 단계는 클릭 시 4.7 복구 옵션 노출.

> 재시도 최적화: 재시도는 `failed` 단계와 그 이후만 다시 실행. 백엔드는 해당 `stepRef`부터 재개하므로 "전체 작업 재실행 불필요" 요구를 충족.

## 4.4 Tool Call Panel

### 노트 내용
AI가 **어떤 도구를, 왜 호출했는지** 보여주는 영역. 포함 정보:

| 정보 | 설명 |
| --- | --- |
| 도구 이름 | 무엇을 호출했나 |
| 호출 목적 | 왜 호출했나 |
| 입력값 | 어떤 파라미터로 |
| 작업 종류 | 조회/생성/변경 |
| 실행 상태 | 진행·성공·실패 |
| 결과 요약 | 무엇을 받았나 |
| 실패 원인 | 왜 실패했나 |
| 재시도 가능 여부 | 다시 할 수 있나 |

> 이건 "신뢰"의 문제다. 결과만 던지면 "어떤 데이터를 보고 그렇게 말했지?"가 남는다. 필요하면 **JSON 그대로** 보여주는 것이 신뢰.

### 상태 매핑
`toolCalls[]` 슬라이스 — 위 8개 정보가 `name / purpose / input / kind / status / resultSummary / error / retryable`에 1:1 대응. 추가로 `rawResult`(JSON 원문), `criticality`(4.7 연계), `attempt`(재시도 횟수).

### 백엔드 미러링
OpenManus의 `tool_call` 이벤트 → `toolCalls`에 항목 append(`status="running"`). `tool_result` → 같은 항목을 `set`으로 갱신(`status="success"`, `resultSummary`, `rawResult`). 검색·조회형 도구의 출처는 동시에 `citations[]`로도 분기(근거 패널용).

### 프론트 렌더링
"Tool Call Panel": 도구별 접이식 카드. 헤더(이름·목적·상태 배지), 본문(입력 파라미터, 결과 요약), [JSON 보기] 토글(rawResult). 실패 시 원인 + [재시도] 버튼(retryable일 때).

## 4.5 Approval Gate

### 노트 내용
단순 읽기면 무방하지만 **외부 시스템 변경**이면 승인이 필요하다.

작업 위험도 등급:

| 등급 | 예시 | 승인 |
| --- | --- | --- |
| Read Only | 데이터 조회, 문서 검색 | 자동 실행 |
| Draft | 이메일·리포트 초안 | 사용자 검토 |
| Internal Write | 내부 티켓 생성, 임시 저장 | 정책에 따라 |
| External Side Effect | 고객 이메일 발송, 캘린더 초대 | **명시적 승인** |
| High Risk | 결제, 삭제, 권한 변경, 배포 | **강한 승인** |

Approval UI에 반영할 것: AI가 하려는 작업 · **영향 대상** · 되돌릴 수 있는지 · 사용 데이터 · 최종 실행 **미리보기** · 승인/수정/취소 버튼.

```text
이메일 124명에게 발송
수신자: [목록 보기]   제목: payment reminder   본문: [미리보기]
되돌리기: 발송 후 취소할 수 없습니다.
[ Edit ]  [ Cancel ]  [ Approve and Send ]
```

> 판단 기준 = "외부 영향". 모든 작업에 승인을 요구하면 UX가 답답하다. 내부 조회·초안 → 승인 불필요 / 이메일·삭제·결제 → 승인 필요.

### 상태 매핑
`approvals[]` 슬라이스 — `riskLevel`(5등급), `action`, `affected`(영향 대상), `reversible`, `preview`, `status`(pending|approved|edited|rejected|expired). `goal.permissions`와 연동하여 **자동 실행 vs 승인 대기**를 결정.

### 백엔드 미러링 (HITL 핵심)
OpenManus 루프가 위험 작업에 도달하면:
1. 작업을 **실행하지 않고** `approvals`에 `pending` 항목을 `set`.
2. `status="waiting_for_approval"`로 전이.
3. 루프를 일시정지하고 사용자 커맨드를 대기.

사용자가 프론트에서 승인하면 **custom command** `approve`(또는 `edit-and-approve` / `reject`)가 백엔드로 전달 → OpenManus가 해당 작업을 실행하고 루프 재개.

> 이 패턴은 2026의 HITL과 정확히 동일: "실행 일시정지 → 사람에게 컨텍스트 제시 → 승인/수정 → 재개". AssistantTransport의 custom command가 이 왕복을 자연스럽게 구현한다.

### 프론트 렌더링
"Approval Gate" 모달/카드: 작업 요약 + 영향 대상 + 되돌리기 가능 여부(경고) + 미리보기 + [Edit][Cancel][Approve and Send]. `riskLevel`이 high_risk면 추가 확인(강한 승인: 체크박스/재입력).

### 승인 경계 자동화 규칙 (의사결정 테이블)
```
permissions.<kind> == "auto"             → 즉시 실행 (approvals 미생성)
riskLevel in {read_only}                 → 즉시 실행
riskLevel in {draft, internal_write}     → 정책 플래그에 따름
riskLevel in {external_side_effect}      → approvals.pending (명시적 승인)
riskLevel in {high_risk}                 → approvals.pending + 강한 승인 UI
```

## 4.6 Streaming Result

### 노트 내용
결과가 **만들어지는 과정**을 보여주되, **신뢰 수준을 이해**하게 만든다. (AI가 한 번에 완성하지 않을 때)

결과 "상태"를 구분하라:

| 상태 | 의미 |
| --- | --- |
| `draft` | 아직 작성 중인 초안 |
| `partial` | **일부 데이터만** 반영된 결과 |
| `verified` | 필요한 데이터와 **검증을 통과**한 결과 |
| `final` | 확정본으로 보여줄 수 있는 결과 |

> 핵심: 결과를 **빨리** 보여주는 것보다, 사용자가 그 결과를 **어느 정도 믿어도 되는지** 알게 하는 것이 중요. 예: Sales API 결과가 없는데 분석을 시작했다면 `final`처럼 보여주면 안 된다.

### 상태 매핑
`messages[].resultStatus`(draft|partial|verified|final) + 보조 메타 `resultMeta`(confidence, pendingInputs). 토큰 스트리밍 자체는 `messages[].content`의 `append-text`로 처리.

### 백엔드 미러링
- 토큰 생성: `state["messages"][-1]["content"] += token` → 자동 `append-text`.
- 신뢰 수준: 아직 도착하지 않은 데이터가 있으면 `resultStatus="partial"`, `resultMeta.pendingInputs`에 미도착 소스 명시. 검증 통과 시 `verified`, 확정 시 `final`로 `set`.

### 프론트 렌더링
메시지 버블에 **신뢰 배지**(draft/partial/verified/final)를 항상 표시. partial이면 안내문:
```text
초안 분석 결과입니다.
marketing 데이터가 도착하면 결론이 업데이트될 수 있습니다.
```
final 이전에는 "확정본 아님"을 시각적으로 구분(예: 점선 테두리, 흐린 배경).

## 4.7 Error Recovery / Partial Failure

### 노트 내용
다양한 실패: `Model Error` · `Tool timeout` · `Permission denied` · `Invalid tool input` · `Network failure` · `Rate limit` …

> 좋은 에러 UI는 실패의 **"위치"와 "다음 행동"**을 알려준다.

좋은 예시:
```text
Marketing API 호출에 실패했습니다.
현재 가능한 선택:
→ Marketing API 다시 시도
→ 데이터 없이 분석 계속
→ 계정 연결 다시하기
→ 작업 취소
```

부분 실패는 전체 중단이 아니다:
```text
✅ Sales API 성공   ❌ Marketing API 실패   ✅ Inventory API 성공
→ sales, inventory 기반으로 분석 완료. marketing은 불러오지 못해 캠페인 영향 분석은 제외.
```

작업 단계의 중요도:

| 중요도 | 의미 | 예시 |
| --- | --- | --- |
| Critical | 없으면 작업 완료 불가 | sales API |
| Important | 없으면 결과 품질 하락 | marketing API |
| Optional | 없어도 결과 생성 가능 | 관련 뉴스 검색 |

### 상태 매핑
`failures[]` 슬라이스 — `type`, `stepRef`/`toolRef`(실패 **위치**), `recoveryOptions`(다음 **행동**), `criticality`. 도구의 `toolCalls[].criticality`와 연동하여 "전체 중단 vs 부분 결과"를 결정.

### 백엔드 미러링
도구 실패 시 `failures`에 항목 추가 + 해당 `steps[].status="failed"`.
- `criticality="critical"` 실패 → `status="error"` 또는 승인 요청(중단).
- `important`/`optional` 실패 → 가능한 만큼 진행하고 `resultStatus="partial"`로 결과 제공.

### 프론트 렌더링
실패 단계 카드에 위치 + 원인 + `recoveryOptions` 버튼들. 부분 실패 시 결과 상단에 "무엇이 빠졌는지" 요약(예: 캠페인 영향 분석 제외).

### 중요도 → 분기 규칙
```
critical failed   → halt or escalate (사용자 결정 요청)
important failed  → degrade gracefully (partial 결과 + 경고)
optional failed   → ignore & continue (조용히 진행, 로그만)
```

## 4.8 Rollback / Undo

### 노트 내용
AI가 외부 시스템을 변경할 수 있다면 **되돌리기 설계 필수**.

| 작업 | 되돌리기 |
| --- | --- |
| 파일 수정, 필드값 변경 | ✅ 되돌릴 수 있음 |
| 이메일 발송 | ⚠️ 정정 메일 (**보상 작업**) |
| 결제, 외부 발송, 영구 삭제 | ❌ 되돌릴 수 없음 |

```text
완료된 작업:  12개 티켓 생성 / 4개 상태 변경 / 2명 알림 발송
되돌릴 수 있는 작업: 티켓 생성 취소, 상태 변경 복구
되돌릴 수 없는 작업: 이미 발송된 알림
```

Undo를 위한 **Action Log** (AI Action 실행 시 프론트가 남김):
```ts
{ actionType, target, previousValue, newValue, timestamp, approvalId }
```
예: `previousValue: todo / newValue: done / inverseAction: restore todo`(되돌리기 = 역연산).

### 상태 매핑
`actionLog[]` 슬라이스 — 노트의 필드(actionType, target, previousValue, newValue, timestamp, approvalId) + `inverseAction` + `reversibility`(reversible|compensable|irreversible).

### 백엔드 미러링
실행이 일어나는 모든 Action에 대해 `actionLog`에 항목을 `set`. `reversibility`는 작업 종류로 결정(필드 수정=reversible, 이메일=compensable, 결제/삭제=irreversible). 되돌리기 커맨드(`undo`) 수신 시 `inverseAction`을 실행하고 결과를 로그에 반영.

### 프론트 렌더링
"Completed Actions" 패널을 3구역으로: 되돌릴 수 있음 / 보상 작업 필요 / 되돌릴 수 없음. reversible 항목에만 [되돌리기] 버튼. irreversible은 비활성 + 사유 표기.

## 4.9 Audit Trail

### 노트 내용
**AI가 수행한 작업의 기록**. 기록할 것: 누가 요청 · 어떤 프롬프트 · 어떤 계획 · 어떤 도구 · 어떤 input · 사용자가 어디서 승인 · rollback 여부.

> 주의: **민감한 raw data는 저장하면 안 된다.** 기록은 "무엇을 했나"의 추적용이지 원본 데이터 보관용이 아니다.

### 상태 매핑
`audit` 슬라이스 — `requestedBy / promptId / planId / toolCallIds / approvalIds / rollbackPerformed`. raw 결과는 참조 ID만 보관(원문은 `rawResult`에 일시적으로만, 영구 저장은 ID/요약).

### 백엔드 미러링
각 단계가 진행될 때 audit 메타를 누적 `set`. 영구 저장(DB)으로 내보낼 때 **민감 raw data를 제외**하고 ID·요약·타임스탬프만 적재.

### 프론트 렌더링
"Audit Trail" 뷰(또는 세션 종료 후 리포트): 요청자 → 프롬프트 → 계획 버전 → 도구 호출 목록 → 승인 지점 → rollback 여부를 타임라인으로. 민감 데이터는 마스킹/요약.

# Part 5. 상태 머신

## "기다림"도 종류가 다르다 — 왜 상태를 잘게 나누는가

노트의 핵심 원칙: 상태를 나누는 기준은 **사용자의 행동**과 **시스템의 복구 방식**이다. 겉보기에 같은 "기다리는 중"이라도 **왜 기다리는지가 다르면 다른 상태**다.

| 상태 | 실제로 일어나는 일 | 사용자에게 주는 의미 |
| --- | --- | --- |
| `planning` | 외부 호출 없이 **AI가 계획을 짜는 중** | "생각 중이에요" |
| `tool_calling` | **외부 API·도구를 실제 호출 중** | "데이터 가져오는 중이에요" |

> 둘 다 스피너만 돌리면 "왜 이렇게 오래 걸리지?" → 이탈. 기다림의 **이유**를 구분하면 이탈률이 크게 준다.

## 전체 상태 전이도

```text
        ┌─────────┐
        │  idle   │
        └────┬────┘
             │ submit goal (add-message)
        ┌────▼────────────┐
        │ prompt_submitted │
        └────┬────────────┘
             │

       └───┬──────────────────┘ └─┘
       └───┬──────────────────┘
           │ done
       ┌───▼────────┐

       └────────────┘

```

## 상태별 UI 신호 (노트의 "기다림 이유 구분"을 실제로)

| status | 화면 신호 | 한 줄 카피 |
| --- | --- | --- |
| `planning` | 계획 스켈레톤 + 점진 표시 | "실행 계획을 세우고 있어요" |

# Part 6. 데이터 흐름

## 커맨드(프론트→백엔드)와 상태 스냅샷(백엔드→프론트)의 왕복

```text
┌────────────────────── Frontend (assistant-ui) ──────────────────────┐
│  Goal Composer / Plan Preview / Approval Gate / ...                  │
│        │  사용자 액션                                                │
│        ▼                                                             │
│  Commands:                                                           │
│    • add-message          (목표 제출)                                │
│    • revise-plan          (계획 수정)        ← custom command        │
│    • approve / reject     (승인/거부)        ← custom command        │
│    • undo                 (되돌리기)         ← custom command        │
│    • cancel               (취소)                                     │
└───────────────────────────────┬─────────────────────────────────────┘
                                 │ POST { state, commands, threadId, ... }
                                 ▼
┌────────────────────── Backend (FastAPI + assistant-stream) ─────────┐
│  run_callback(controller):                                          │
│    1) for cmd in commands: 상태 반영 (예: add-message → messages)    │
│    2) OpenManus 루프 실행:                                          │
│         plan        → state["plan"]        (set)                     │
│         step start  → state["steps"]       (append/set)             │
│         tool_call   → state["toolCalls"]   (append)                 │
│         tool_result → state["toolCalls"]   (set) + citations        │
│         risk action → state["approvals"]   (set) + 루프 일시정지     │
│         token       → state["messages"][-1]["content"] (append-text)│
│         failure     → state["failures"]    (append) + steps failed  │
│         action      → state["actionLog"]   (set)                    │
└───────────────────────────────┬─────────────────────────────────────┘
                                 │ stream  aui-state:[{op}, ...]
                                 ▼
              converter(state) → assistant-ui messages
              useAssistantTransportState(selector) → 각 패널
```

## 커맨드 수명주기 (idle ↔ sending)

- 런타임은 **idle**(요청 없음)과 **sending**(요청 진행 중)을 오간다.
- idle에서 새 커맨드 → 즉시 전송. sending 중이면 → 큐잉 후 현재 요청 완료 시 전송.
- 승인/취소/되돌리기 같은 사용자 액션도 이 수명주기를 따른다.

## 취소·연결 끊김 처리

- 응답 스트림이 일찍 닫히면 `controller.is_cancelled`가 set → OpenManus 루프가 `while not controller.is_cancelled` 조건으로 깔끔히 종료.
- 중요한 정리(외부 트랜잭션 롤백 등)는 `finally` 블록에.
- `onCancel`에서 `updateState`로 클라이언트 상태만 `status="cancelled"`로 갱신(서버 요청 없이).

| `ToolCallPanel` | `s => s.toolCalls` | 4.4 |
| `ApprovalGate` | `s => s.approvals` | 4.5 |
| `ResultMessage` | converter messages + `s => s.resultMeta` | 4.6 |
| `CitationPanel` | `s => s.citations` | (근거) |

```tsx
// 예시: 타임라인은 steps만 구독
import { useAssistantTransportState } from "@assistant-ui/react";

function AgentTimeline() {
  const steps = useAssistantTransportState((s) => s.steps);
  return (
    <ol>
      {steps.map((st) => (
        <li key={st.id} data-status={st.status}>
          {icon(st.status)} {st.label}
          {st.status === "failed" && <RecoveryActions stepId={st.id} />}
        </li>
      ))}
    </ol>
  );
}
```

```tsx
// 예시: 승인 게이트는 custom command를 발행
import { useAssistantTransportSendCommand } from "@assistant-ui/react";

function ApprovalGate({ approval }) {
  const send = useAssistantTransportSendCommand();
  return (
    <div>

      <button onClick={() => send({ type: "approve", data: approval.id })}>

      </button>

  );
}
```

# Part 8. 측정/관측

## 노트가 제시한 지표 (굵게 = 우선 구현)

- **AgentRunCompletionRate**
- **Approval rate**
- **Tool failure rate**
- **Rollback rate**
- user intervention rate
- **Time to first useful progress** = (AI가 실제 작업 시작 인지 시점) − (요청 제출 시점)
- time to completion
- unsafe action blocked rate

## 지표 → 제품 판단

| 지표 | 값이 나쁠 때의 의미 |
| --- | --- |
| AgentRunCompletionRate ↓ | AI 계획이 사용자 의도와 안 맞음 |
| Tool failure rate ↑ | 모델이 아니라 **도구 연동** 문제가 경험을 망침 |
| Rollback rate ↑ | 실행 정확도 또는 **승인 경계** 부족 |
| Time to first useful progress ↑ | "멈췄다"고 느낌 → Partial Progress UI·타임라인 필요 |

## 상태 트리에서 지표를 어떻게 뽑는가

각 지표는 상태 슬라이스에서 직접 파생된다. → **별도 계측 코드 없이 상태에서 계산**.

| 지표 | 계산식 (상태 기반) |
| --- | --- |
| Time to first useful progress | `metrics.firstUsefulProgressAt − metrics.submittedAt` |
| time to completion | `metrics.completedAt − metrics.submittedAt` |
| Approval rate | `approvals[approved] / approvals[*]` |
| Tool failure rate | `toolCalls[failed] / toolCalls[*]` |
| Rollback rate | `actionLog[undone] / actionLog[*]` |
| user intervention rate | (revise-plan + edit-approve + undo) / runs |
| unsafe action blocked rate | `approvals[rejected & high_risk] / approvals[high_risk]` |

> `firstUsefulProgressAt`는 첫 번째 `steps[].status="running"`이 찍히는 순간으로 정의 → 노트의 "AI가 실제 작업을 시작했다고 인지한 시점"과 일치.

# Part 9. 구현 로드맵

> 노트의 안전장치 강조 순서(승인·복구·되돌리기·감사)를 반영해, **신뢰·안전 슬라이스를 뒤로 미루지 않는다.**

## Phase 0 — 뼈대 (1일차)
- `npx assistant-ui@latest create -t minimal`
- FastAPI + `assistant-stream` 엔드포인트, mock `run_callback`로 `messages`만 스트리밍.
- 상태 트리 타입(부록 A) 선언 + module augmentation으로 타입 안전성 확보.
- **완료 기준**: 채팅 한 줄이 왕복하고, `messages.content`가 `append-text`로 스트리밍됨.

## Phase 1 — Plan + Timeline (과정 가시화)
- OpenManus 연결, `plan`·`steps` 슬라이스 미러링.
- `PlanPreview`, `AgentTimeline` 컴포넌트.
- 상태 머신 `planning`/`tool_calling` 구분 도입(스피너 카피 분리).
- **완료 기준**: "무엇을 하고 있나"가 화면에 단계로 보임.

## Phase 2 — Tool Call + Citations (신뢰의 1차)
- `toolCalls` 미러링(목적·입력·결과·JSON), `citations` 분기.
- `ToolCallPanel`, `CitationPanel`.
- **완료 기준**: 각 결론에 "어떤 도구/출처에서 나왔는지" 추적 가능.

## Phase 3 — Approval Gate (HITL)
- `approvals` 슬라이스 + custom command(`approve`/`reject`/`edit-and-approve`).
- 위험도 등급 의사결정 테이블 적용, 루프 일시정지/재개.
- `ApprovalGate` 모달, 강한 승인 UI(high_risk).
- **완료 기준**: 외부 영향 작업이 승인 없이는 실행되지 않음.

## Phase 4 — Result 신뢰 수준 + Error Recovery
- `resultStatus`(draft/partial/verified/final) + 신뢰 배지.
- `failures` 슬라이스 + 중요도(critical/important/optional) 분기 + 복구 옵션.
- 부분 실패 시 graceful degradation.
- **완료 기준**: 실패해도 가능한 만큼 결과 제공 + 신뢰 수준 명시.

## Phase 5 — Rollback/Undo + Audit + 지표
- `actionLog`(역연산/보상/불가) + `undo` 커맨드.
- `audit` 슬라이스(민감 raw 제외) + 영구 저장.
- Part 8 지표를 상태에서 계산 + 대시보드.
- **완료 기준**: 한 실행의 전 과정을 되돌리거나 감사할 수 있음.

# Part 10. 보강 축

> 노트가 낡아서가 아니라, **수록 범위에서 빠진 2026 최신 축**. 위 로드맵에 얹으면 완성도가 올라간다.

## 10.1 Autonomy Dial (자율성 단계)
신뢰는 이분법이 아니다. 사용자가 task별로 "suggest only → act and notify me after"를 조절하게 한다(전역 on/off가 아니라 스펙트럼).
- 상태: `goal.permissions`를 단순 auto/require가 아니라 `suggest_only | ask_each_time | act_and_notify | full_auto`로 확장.
- UI: task 유형별 자율성 슬라이더.

## 10.2 Checkpoint Resume (상태 영속화/재개)
페이지 새로고침·장시간 대기 후에도 정확한 체크포인트에서 재개. (HITL의 인프라 요건: 상태 저장은 non-negotiable)
- 구현: `resumeApi` + `resumeRun`. 백엔드가 생성 중이면 마지막 메시지 ID로 재연결.
- 상태: `threadId`별 스냅샷 영속화(이미 상태 트리가 직렬화 가능하므로 자연스럽게 지원).

## 10.3 Human-on-the-loop (성숙도 사다리)
"모든 단계 검토(human-in-the-loop)"에서 "예외만 통치(governing by exception)"로 이동.
- 상태: 승인 임계값을 신뢰도 기반으로(예: confidence < 0.9일 때만 approval 생성).
- 효과: 저위험·고신뢰 작업은 자동, 경계 사례만 사람에게.

## 10.4 Evidence Pack (리뷰어 인터페이스)
리뷰어가 ~15초 안에 결정하도록 **상태 트레이스·신뢰도·정책 근거를 압축**해 제시.
- 구현: 승인/리뷰 시점에 `toolCalls` 요약 + `resultMeta.confidence` + 관련 `citations`를 한 카드로 묶어 노출.
- 위치: Approval Gate와 Audit Trail 사이의 "왜 이 결정이 안전한가"를 한눈에.

# Part 11. 보안 및 신뢰성 (Security & Reliability)

> Agentic UI는 AI가 실제로 외부 시스템을 변경하는 구조이므로, 보안은 기능이 아니라 **전제 조건**이다.

## 11.1 승인 권위의 서버화 (Server-side Authority)
- **문제**: AssistantTransport는 `state`와 `commands`를 프론트에서 통째로 받아 백엔드로 전달한다. 클라이언트가 조작한 `riskLevel`이나 `affected`를 믿고 실행하면 심각한 취약점이 된다.
- **해결**: 프론트의 `approve` 커맨드는 `approvalId`만 전달해야 한다. 백엔드는 **서버가 보관한 권위 사본**과 대조해 위험도와 미리보기를 재검증한 후 실행한다.

## 11.2 상태 마스킹 및 시크릿 차단
- **문제**: 상태 트리가 통째로 프론트로 스트리밍되므로, 도구 호출 결과(`rawResult`, `input`)나 미리보기(`preview`)에 API 키나 PII(개인식별정보)가 섞이면 즉시 유출된다.
- **해결**: 
  - 클라이언트가 보내는 `config.apiKey` 경로는 봉쇄하고 서버 측 환경변수로 고정한다.
  - 백엔드에서 상태를 스트리밍하기 직전, 민감 데이터를 마스킹(Redaction)하는 파이프라인을 필수 적용한다.

## 11.3 프롬프트 인젝션 방어 및 정책 엔진
- **문제**: 도구가 가져온 외부 문서(검색 결과 등)에 악성 명령이 섞여 `high_risk` 작업을 트리거할 수 있다.
- **해결**: 
  - `forbiddenActions`를 단순 UI 토글이 아니라 **서버 측 정책 엔진의 Deny 규칙**으로 강제한다.
  - `external_side_effect` 이상 작업은 어떤 경우에도 승인 게이트를 우회할 수 없도록 백엔드에서 차단한다.

## 11.4 멱등성(Idempotency)과 동시성
- **동시성**: 같은 thread에 두 탭이 열리면 상태 충돌이 발생한다. 백엔드에서 thread 락을 걸거나 Optimistic Concurrency(버전/ETag)를 적용한다.
- **멱등성**: 네트워크 지연으로 `approve`나 `undo` 커맨드가 중복 전송될 수 있으므로, 부수효과가 있는 커맨드는 Idempotency Key로 중복 실행을 방지한다.

# 부록 A. 전체 TypeScript 상태 타입

```ts
// assistant.config.ts — module augmentation으로 상태 타입 주입
import "@assistant-ui/react";

declare module "@assistant-ui/react" {
  namespace Assistant {
    interface ExternalState {
      threadId: string;
      status: AgentStatus;
      messages: ChatMessage[];
      goal: Goal;
      plan: Plan;
      steps: Step[];
      toolCalls: ToolCall[];
      approvals: Approval[];
      resultMeta: ResultMeta;
      failures: Failure[];
      actionLog: ActionLogEntry[];
      audit: Audit;
      citations: Citation[];
      metrics: Metrics;
    }
  }
}

type AgentStatus =
  | "idle" | "prompt_submitted" | "planning" | "plan_ready"
  | "waiting_for_approval" | "tool_calling" | "streaming"
  | "partial_failure" | "completed" | "cancelled" | "error";

interface ChatMessage {
  id: string;
  role: "user" | "assistant";
  content: string;
  // goal 필드는 최상위 스냅샷으로 이관됨
  resultStatus?: "draft" | "partial" | "verified" | "final";
}

interface Goal {
  objective: string;
  scope: string[];
  dataSources: string[];
  outputFormat: string[];
  forbiddenActions: string[];
  permissions: Record<"read" | "create" | "update" | "delete" | "external", AutonomyLevel>; // ToolCall kind와 네임스페이스 통일
}
type AutonomyLevel =
  | "suggest_only" | "ask_each_time" | "act_and_notify" | "full_auto"
  | "auto" | "require_approval"; // 하위호환

interface Plan {
  status: "drafting" | "ready" | "approved" | "revised";
  editable: boolean;
  steps: { id: string; text: string }[];
}

interface Step {
  id: string;
  planRef?: string;
  label: string;
  status: "pending" | "running" | "done" | "failed" | "skipped";
  toolCallId?: string;
  startedAt?: number;
  endedAt?: number;
}

interface ToolCall {
  id: string;
  name: string;
  purpose: string;
  kind: "read" | "create" | "update" | "delete" | "external";
  criticality: "critical" | "important" | "optional";
  input: unknown;
  status: "pending" | "running" | "success" | "failed";
  resultSummary: string | null;
  rawResult: unknown | null;
  error: string | null;
  retryable: boolean;
  attempt: number;
}

interface Approval {
```

## 맺음 — 이 문서의 사용법

1. **노트의 9단계 = 상태 트리의 9개 슬라이스**라는 1:1 대응이 이 설계의 척추다.
2. 각 슬라이스는 **백엔드에서 미러링**(부록 B)되고 **프론트에서 구독**(Part 7)된다.
3. "기다림의 이유"를 구분하는 **상태 머신**(Part 5)이 사용자 신뢰와 이탈률을 좌우한다.
4. 안전장치(승인·복구·되돌리기·감사)는 **뒤로 미루지 않고** Phase 3~5에서 반드시 구현한다.
5. 지표(Part 8)는 별도 계측이 아니라 **상태에서 파생**된다.
6. Part 10의 4개 축으로 2026 수준의 완성도까지 끌어올린다.
